The World of Class and Objects

We have been hearing that Ruby is an Object Oriented Programming language, but what exactly does this mean? Perhaps, we should start from understanding the concept behind an OOP language, and for this it will be helpful to understand first what it is not, or how does it differ with the “traditional” way of programming. In a Non-OOP, the approach used to be focused on the procedures, which needed to get all the data in, process it and return it for further processing. The data was treated as a passive element that could be easily accessed and modified by a powerful procedure.

In a OOP, the approach is completely different. Instead of trying to deal with the logic of the procedures, we focus in identifying and defining the data objects that we are going to be managing. We specify their states (variables), the way they behave (methods), and the way they relate/respond to each others(data-modeling). We create general classes that set the standard or basic characteristics of the objects it represents, the kind of data they will hold, the way they will behave internally and externally, and most important which set of data will be accessible to others and which not.

All these specifications, allows us to create class instances (objects) that are full of well specified data, and that are much easier to understand, especially if you are not the original developer of the code, or just a contributor.

This concept of class and objects can be easily understood if we make a metaphor to visualize it, so let’s suppose that our name is George and we are thinking in writing a short novel that is based in the story of some families that ruled the land… and we will call this novel:

The first thing we would like to do is to define the characteristics that these families will share in common, and on the same time differ from each other, as well as how they will predictably behave. In order to differ them from other elements of our story we will give them the class of House.

So everytime we want to create a family, an instance from this class house , we will know some things for fact:

  • They will have a name…
  • which will be their last-name
  • They will live in Westeros…
  • but we have to specify their location
  • They have a sigil…
  • but we have to design one.
  • They will have a lord…
  • but he could change from time to time.
  • They will have words…
  • but we have to decide on them

The houses will be famous for this characteristics, but they will also keep some secrets inside the Family, some local variables, and some particular methods of reacting to events or circumstances. For example, in some occasions, they will return their strings or words. The action itself will be the same, but the argument that we will pass to each family will be different, and so will be their words in return.

Now that our rules are set, we can create our first family, for instance, we could start with the Lannisters:

  • They will have a name…
  • Lannister
  • They will live in Westeros…
  • The Westerlands
  • They have a sigil…
  • A golden lion rampant on a crimson field.
  • They will have a lord…
  • Tywin Lannister.
  • They will have words…
  • “Hear Me Roar!

We could continue like this and create the Major Houses, or even more, because on Ruby, the possibility of creating new classes or objects is infinite. And hopefully now, that we understand its bases, we can start coding and get ready, because Winter is coming!
